```

│
├── backend/                 # 백엔드 팀 문서
│   ├── api_specs/          # API 명세서
│   ├── database/           # DB 스키마 문서
│   ├── guides/             # 개발 가이드
│   └── tasks/              # 작업 명세
│
├── mobile/                  # 모바일 팀 문서
│   ├── features/           # 기능별 명세
│   ├── ui_specs/           # UI/UX 명세
│   ├── guides/             # 개발 가이드
│   └── tasks/              # 작업 명세

    └── templates/          # 문서 템플릿
```

## 📋 문서 카테고리

### PM 문서 (`pm/`)

#### 기능 명세서 (`specs/`)
- 새로운 기능 개발을 위한 상세 명세서
- 비즈니스 요구사항, 기술 요구사항, 수용 기준

#### 인수인계 문서 (`handoff/`)
- 작업 완료 후 인수인계 문서
- 변경 사항, 테스트 결과, 다음 단계

#### 회의록 (`meetings/`)
- 주간 회의, 스프린트 리뷰, 기획 회의 기록

#### 릴리즈 노트 (`release_notes/`)
- 버전별 릴리즈 노트
- 변경 사항, 버그 수정, 알려진 이슈

---

### 백엔드 문서 (`backend/`)

#### API 명세서 (`api_specs/`)
- RESTful API 엔드포인트 명세
- 요청/응답 형식, 에러 코드
- 예: `POST_api_payment_request.md`

#### 데이터베이스 (`database/`)
- 테이블 스키마
- ERD 다이어그램
- 마이그레이션 가이드

#### 개발 가이드 (`guides/`)
- 코딩 컨벤션
- 테스트 가이드
- 배포 가이드

#### 작업 명세 (`tasks/`)
- 백엔드 작업 상세 명세
- 예: `TASK-001_티켓검증API구현.md`

---

### 모바일 문서 (`mobile/`)

#### 기능별 명세 (`features/`)
- 기능별 상세 구현 명세
- 화면 플로우, 상태 관리
- 예: `FEATURE_티켓상세페이지.md`

#### UI/UX 명세 (`ui_specs/`)
- 화면 디자인 명세
- 위젯 구조
- 사용자 인터랙션

#### 개발 가이드 (`guides/`)
- Flutter 코딩 컨벤션
- 상태 관리 패턴
- 테스트 가이드

#### 작업 명세 (`tasks/`)
- 모바일 작업 상세 명세
- 예: `TASK-002_결제화면개발.md`

---

### 공통 문서 (`shared/`)

#### 아키텍처 (`architecture/`)
- 시스템 아키텍처
- 데이터 플로우
- 보안 설계

#### 워크플로우 (`workflow/`)
- 개발 프로세스
- Git 브랜치 전략
- 코드 리뷰 프로세스

#### 템플릿 (`templates/`)
- 문서 작성 템플릿
- 기능 명세서 템플릿
- API 명세서 템플릿

---

## 📝 문서 작성 규칙

### 파일명 규칙
```
# 날짜 포함 (회의록, 릴리즈 노트)
2026-02-09_주간회의.md
RELEASE_v1.0.0_2026-03-01.md

# 기능/작업 (명세서, 작업 문서)
FEATURE_티켓검증시스템.md
TASK-001_QR코드스캔구현.md
API_POST_payment_request.md

# 가이드 (일반 문서)
코딩컨벤션.md
배포가이드.md
```

### 문서 작성 가이드
1. **명확한 제목**: 문서 내용을 한눈에 파악 가능
2. **날짜 표시**: 작성일, 최종 수정일 명시
3. **버전 관리**: 주요 변경 시 버전 업데이트
4. **상태 표시**: ✅ 완료, 🚧 진행중, ⏳ 예정
5. **링크 활용**: 관련 문서 상호 참조

## 🔄 문서 업데이트 프로세스

### 1. 문서 생성
- PM이 초안 작성
- 해당 팀에 리뷰 요청

### 2. 검토 및 승인
- 개발팀이 기술적 검토
- 피드백 반영

### 3. 최종 승인
- PM 최종 승인
- 문서 상태를 "✅ 승인됨"으로 변경

### 4. 배포
- 개발팀에 전달
- 작업 시작

### 5. 업데이트
- 변경 사항 발생 시 문서 업데이트
- 버전 업데이트 및 히스토리 기록
